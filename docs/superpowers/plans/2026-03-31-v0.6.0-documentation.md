# Profiles (Multi-Instance Hermes)

Profiles allow you to run multiple isolated Hermes instances from a single installation. Each profile gets its own config, memory, sessions, skills, and gateway service. This is useful for separating work and personal use, running multiple bots with different identities, or experimenting with different configurations without affecting your primary setup.

Introduced in **v0.6.0 (v2026.3.30)**.

---

## Creating and Switching Profiles

```bash
hermes profile create <name>     # create a new profile
hermes profile list              # list all profiles
hermes profile switch <name>     # set a profile as default
hermes profile delete <name>     # delete a profile and its data
hermes profile export <name>     # export profile to a tar archive
hermes profile import <file>     # import profile from archive
hermes profile rename <old> <new> # rename a profile
```

Each profile gets:
- Its own `HERMES_HOME` directory (e.g. `~/.hermes-profiles/work/`)
- Its own `config.yaml`, `.env`, memory, sessions, skills, and cron jobs
- Its own gateway service (separate systemd/launchd unit when running `hermes gateway start`)
- A generated CLI wrapper script at `~/.local/bin/hermes-<name>` for quick access

## Running a Specific Profile

Use the `-p` / `--profile` flag:

```bash
hermes -p work chat              # open chat as the "work" profile
hermes -p work gateway start     # start the gateway for the "work" profile
hermes-work chat                 # equivalent via wrapper script (tab-completion supported)
```

## Token Lock Isolation

Profiles enforce token lock isolation: two profiles cannot use the same bot credential (API key, Telegram token, etc.) simultaneously. If you attempt to start a gateway using a credential that is already active in another profile, Hermes will raise a clear error message rather than silently competing.

## Exporting and Sharing Profiles

```bash
hermes profile export work --output work-profile.tar.gz
hermes profile import work-profile.tar.gz
```

Exported archives include config, memory, skills, and sessions. API keys in `.env` are stripped from the export by default.

## Profile Directory Layout

Each profile's home directory mirrors the standard `~/.hermes/` layout:

```text
~/.hermes-profiles/<name>/
├── config.yaml
├── .env
├── auth.json
├── SOUL.md
├── memories/
├── skills/
├── cron/
├── sessions/
└── logs/
```

The active profile's home directory is exposed as `HERMES_HOME` in all Hermes processes.
```

## Docker Container

Official Docker support was added in **v0.6.0 (v2026.3.30)**. The `Dockerfile` in the repository builds a container that supports both CLI and gateway modes with volume-mounted config.

### Build the Image

```bash
cd ~/.hermes/hermes-agent   # or wherever the repo is cloned
docker build -t hermes-agent .
```

### Run in CLI Mode

```bash
docker run -it --rm \
  -v ~/.hermes:/root/.hermes \
  hermes-agent hermes chat
```

### Run in Gateway Mode

```bash
docker run -d \
  --name hermes-gateway \
  -v ~/.hermes:/root/.hermes \
  -p 8080:8080 \
  hermes-agent hermes gateway start
```

The container reads config and writes sessions/logs to the mounted `~/.hermes` directory, so your data persists between container restarts.

### Using Podman

Substitute `podman` for `docker` in all commands above — the Dockerfile is OCI-compatible.
```

## MCP Server Mode (v0.6.0)

Hermes can expose its own conversations and sessions as an MCP server, letting any MCP-compatible client (Claude Desktop, Cursor, VS Code, etc.) browse and search your Hermes sessions.

### Starting the MCP Server

```bash
# stdio transport (for Claude Desktop, Cursor, VS Code MCP config)
hermes mcp serve

# Streamable HTTP transport
hermes mcp serve --transport http --port 8765
```

### Available Tools (Exposed to MCP Clients)

When a client connects, Hermes exposes:

| Tool | Description |
|------|-------------|
| `list_sessions` | List all Hermes sessions with metadata |
| `read_session` | Read full message history for a session |
| `search_sessions` | Full-text search across all sessions |
| `list_attachments` | List media attachments for a session |

### Claude Desktop Configuration

```json
{
  "mcpServers": {
    "hermes": {
      "command": "hermes",
      "args": ["mcp", "serve"]
    }
  }
}
```

### HTTP Transport Configuration

```json
{
  "mcpServers": {
    "hermes": {
      "url": "http://localhost:8765/mcp"
    }
  }
}
```

### Dynamic Tool Discovery

Hermes also responds to `notifications/tools/list_changed` events from MCP servers it connects to, picking up new tools from running servers without requiring a restart.
```

### Mounting Skills and Credentials on Remote Backends (v0.6.0)

When running terminal commands in Modal or Docker containers, Hermes can now mount your local skill directories and credential files into the container so remote sessions have the same capabilities as local execution.

**Skills on remote backends:**

Configure which skill directories to mount in `config.yaml`:

```yaml
terminal:
  backend: modal    # or docker
  remote:
    mount_skills: true   # mounts ~/.hermes/skills/ into container
```

**Credential files on remote backends:**

```yaml
terminal:
  backend: modal
  remote:
    mount_credentials:
      - ~/.hermes/.env
      - ~/.config/gcloud/application_default_credentials.json
```

Files are cached by mtime and size — Hermes only re-uploads when the local file changes.
```

Full Feishu (飞书) and Lark gateway adapter, added in **v0.6.0 (v2026.3.30)** ([#3799](https://github.com/NousResearch/hermes-agent/pull/3799), [#3817](https://github.com/NousResearch/hermes-agent/pull/3817)).

## Features

- Rich message cards

- Image and file attachments

gateway:

FEISHU_APP_ID=cli_xxxxxxxxxxxx
FEISHU_APP_SECRET=your_app_secret
FEISHU_VERIFICATION_TOKEN=your_verification_token
FEISHU_ENCRYPT_KEY=your_encrypt_key   # optional, for encrypted events

For Lark (international), the same env vars apply — the adapter auto-detects the domain from the App ID prefix.

## Gateway Setup

```bash
hermes gateway setup feishu
hermes gateway start
```

## Message Cards

Hermes responses are sent as Feishu message cards when the response is long or contains structured content. Short text replies are sent as plain text messages.

## Webhook Mode (v0.6.0)

By default, the Telegram adapter uses long-polling. For production deployments behind a reverse proxy, webhook mode provides faster response times and lower resource usage.

### Setup

1. Your gateway must be accessible at a public HTTPS URL.
2. Set in `~/.hermes/.env`:

```
TELEGRAM_WEBHOOK_URL=https://yourserver.example.com/webhook/telegram
```

3. Restart the gateway — Hermes registers the webhook with Telegram automatically.

To switch back to polling, remove `TELEGRAM_WEBHOOK_URL`.

---

## Group Mention Gating (v0.6.0)

Control when the bot responds in group chats via `TELEGRAM_GROUP_RESPOND_MODE` in `.env`:

| Value | Behavior |
|-------|----------|
| `always` (default) | Respond to every message in the group |
| `mention` | Respond only when the bot is @mentioned |
| `regex:<pattern>` | Respond when the message matches the regex pattern |

**Examples:**
```
# Only respond when @mentioned
TELEGRAM_GROUP_RESPOND_MODE=mention

# Respond when message contains "hermes" (case-insensitive)
TELEGRAM_GROUP_RESPOND_MODE=regex:(?i)hermes
```
```

## Multi-Workspace OAuth (v0.6.0)

A single Hermes gateway can now connect to multiple Slack workspaces simultaneously via an OAuth token file ([#3903](https://github.com/NousResearch/hermes-agent/pull/3903)).

### Setup

Create a token file at `~/.hermes/slack_tokens.json`:

```json
{
  "workspaces": {
    "T0123456789": {
      "bot_token": "xoxb-workspace-one-token",
      "name": "Work Workspace"
    },
    "T9876543210": {
      "bot_token": "xoxb-workspace-two-token",
      "name": "Personal Workspace"
    }
  }
}
```

Configure Hermes to use the token file:

```yaml
# config.yaml
gateway:
  platforms:
    slack:
      token_file: ~/.hermes/slack_tokens.json
```

Hermes resolves the correct bot token dynamically from the workspace ID in each incoming event, so all workspaces share a single gateway process.

### Single-Workspace (Legacy)

The existing `SLACK_BOT_TOKEN` env var still works for single-workspace setups.
```

## Enabling and Disabling Plugins (v0.6.0)

You can enable or disable plugins without removing them from the plugins directory:

```bash
hermes plugins enable <plugin-name>    # enable a disabled plugin
hermes plugins disable <plugin-name>   # disable without deleting
hermes plugins list                    # show all plugins with enabled/disabled status
```

Disabled plugins are not loaded at startup, so their tools and hooks are not available to the agent. The plugin directory and files are preserved.

This is useful for temporarily disabling a plugin that conflicts with another, or for testing a new plugin without committing to always loading it.

## Message Injection (v0.6.0)

Plugins can inject messages into the conversation stream on behalf of the user via `ctx.inject_message()`. This allows plugins to:

- Inject context or reminder messages before the agent processes a user message
- Trigger agent actions based on background events (timers, file watchers, etc.)
- Respond to external webhooks by inserting messages into a running conversation

### Interface

```python
# In a hook handler (e.g., pre_llm_call or a custom command handler)
async def my_hook(ctx):
    await ctx.inject_message(
        content="Reminder: the user's focus timer has elapsed.",
        role="user"   # or "system"
    )
```

Injected messages appear in the conversation transcript and are visible in session exports.

Contributed by @winglian ([#3778](https://github.com/NousResearch/hermes-agent/pull/3778)).
```

### Fallback Provider Chain (v0.6.0)

Configure an ordered list of fallback providers. When the primary provider returns errors or is unreachable, Hermes automatically tries the next provider in the chain:

```yaml
model:
  provider: anthropic
  name: claude-sonnet-4.6

fallback_providers:
  - provider: openrouter
    name: anthropic/claude-sonnet-4.6
  - provider: nous
    name: hermes-3-llama-3.1-70b
```

Hermes tries providers in order and stops at the first successful response. Failed providers are logged at INFO level.

Use this for:
- High-availability setups where one provider may be down
- Rate limit recovery (failover to a backup provider when hitting 429s)
- Cost optimization (primary: cheap provider, fallback: premium)
```

## Fallback Provider Chain (v0.6.0)

The `fallback_providers` key in `config.yaml` defines an ordered list of providers to try when the primary provider fails. See [Configuration → Fallback Provider Chain](./configuration.md) for the full schema.

Hermes triggers fallback on:
- Network errors and timeouts
- HTTP 5xx errors
